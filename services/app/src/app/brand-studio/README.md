# Brand Studio

Internal tool to experiment with the project's design tokens and apply the
chosen theme to the running app. Same code in dev and prod — gating is on
**authenticated SUPERUSER role**, not on `NODE_ENV`.

## Access

- URL: `/brand-studio` (e.g. `https://evo.heuresys.com/brand-studio`).
- Anonymous → redirect to `/login?callbackUrl=/brand-studio`.
- Authenticated, role ≠ `SUPERUSER` → 404.
- Authenticated, role = `SUPERUSER` → wizard renders.
- Server Actions enforce the same role check (defense in depth) — direct API
  calls without a SUPERUSER session are rejected.

## Workflow

1. Sign in as a `SUPERUSER` account (`sysadmin`) and open
   `/brand-studio`.
2. Configure the wizard across the 10 steps (brand, colors, modes, typography,
   spacing, motion, effects, density, iconography, export).
3. **Preview on site** — writes a short-lived cookie with the rendered token
   CSS. Reload any page (e.g. `/dashboard`) to see the theme applied. The
   preview is per-browser and lives 24h or until cleared.
4. **Clear preview** — removes the cookie.
5. **Apply to project** — writes the rendered token CSS to
   `services/app/src/styles/active-theme.css`. That file is imported by the
   root layout, so a reload (or HMR) makes the theme the new default for every
   user of the app. The file header records the applying SUPERUSER username
   and timestamp for audit.
6. **Reset draft** — wipes the localStorage draft and restarts from defaults.

The wizard's own **Export theme** button still produces the 5-file design-token
bundle (`tokens.css`, `tailwind.config.js`, `tokens.json`, `figma-tokens.json`,
`theme.tsx`) as browser downloads, unchanged.

## Persistence

| Layer        | Where                                                       | TTL              |
| ------------ | ----------------------------------------------------------- | ---------------- |
| Draft        | `localStorage["heuresys.theme-studio.draft"]` (per-browser) | until cleared    |
| Preview      | cookie `heuresys-theme-preview` (per-browser)               | 24h              |
| Active theme | `services/app/src/styles/active-theme.css` (committed)      | until next Apply |

## Files

```
services/app/src/
├── app/brand-studio/
│   ├── page.tsx              # Server: auth + role gate, mounts client
│   ├── BrandStudioClient.tsx # Client: wizard + preview/apply controls
│   ├── actions.ts            # Server Actions: write css, set/clear cookie
│   └── README.md
├── app/_components/
│   └── ThemePreviewInjector.tsx  # Client: reads cookie, applies <style>
├── app/layout.tsx            # Imports active-theme.css + injector
└── styles/
    └── active-theme.css      # Generated by Apply
```

## Security notes

- Role gate (`SUPERUSER`) enforced at both the page level and inside every
  Server Action.
- CSS payloads are size-capped (8 KB) and reject `<script>` / `</style>`-like
  sequences in both the Server Action and the client injector.
- The injector uses `style.textContent` (not `innerHTML`) — CSS is treated as
  text, not parsed as HTML.
- File writes target a fixed path (`src/styles/active-theme.css`); no user
  input controls the destination.
- Each Apply is logged in the generated CSS header (username + timestamp).
