## Open questions

- **Repo GitHub: nome?** Il v1 è `github.com/heuresys/heuresys.com.evo` (anche se confusamente chiamato così, è il v1 originale). Per il `.evo` rebuild: `heuresys-evo` (semplice), `heuresys-platform-v2` (più descrittivo), `heuresys.com.evo.v2` (consistency confusa)?
- **Quale frontend bootstrappare per primo: `services/marketing` o `services/app`?** Marketing è più semplice (SSG, no auth, no DB), app è più valuable (la dashboard SaaS è il prodotto core). Iniziare da marketing per validate Next.js 16 + packages/ui + tunnel DB lavoro, poi app. Oppure attaccare app direttamente.
- **WSL2 + Postgres locale dev nativo, o continuare con tunnel SSH?** L'utente preferiva nativo Windows ma timeva divergenza. Tunnel risolve il problema senza installare nulla. Decidere se rimane scelta strategica o se in futuro vorrà WSL2 anche.
- **Migration 221 (TOTP plain drop): applicare nel `.evo` o saltare?** Richiede TOTP backfill (1 user con plain `totp_secret` e `totp_secret_encrypted=NULL`). Decisione: skip nella baseline `.evo` (commit `fe3cf1a`). Riconsiderare quando iniziamo l'auth `.evo`?
- **Dual-CI: workflow separati per v1 vs `.evo`?** I `.github/workflows/*.yml.example` del `.evo` sono solo stub. Quando push GitHub, occorre coordinare i workflow CI per non confliggere col v1 (path-based filtering).
