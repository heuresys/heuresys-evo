## Cross-cutting patterns identificati

1. **Documentation maturity > Implementation maturity**: il pattern dominante è "documento detailed, implementazione partial". Esempi: ADR-0006 servizi fantasma, ADR-0002 testcontainers `Proposed`, ADR-0011 coverage threshold non enforced, ADR-0012 CSP enforce mai flippato, manual a11y checklist tutti ⏳, claim AAA solo axe-based. Acquirer deve "scontare" la documentazione in proporzione al gap implementation.

2. **RLS coverage policy ma test bypass = 0**: P1+P5 multi-tenant è il cuore della value proposition, ma nessun test simula attacker cross-tenant. Trio H13 (RLS bypass test) + H4 (audit log adoption) + H5 (RBP UI gap) è il **vero rischio compliance** della piattaforma.

3. **Founder bus factor estremo**: 314 commit Spen-Zosky / 335 totali = 94%. `incident-runbook` esplicita "Enzo è single point of human failure". Per acquirer = retention package vincolante + onboarding immediato L2 SRE/Senior FullStack pre-day-one.

4. **Brand workstream maturo > codice frontend immaturo**: paradosso. Brand identity v0 shipped + Brand Book completo + design system Cantiere B v2 ricco (~180 component, 84 stories), ma frontend services/app manca pattern essenziali (boundary RSC, next/image, RBP UI gate). Asset acquisibile ma non integrato.

5. **Velocity AI-assisted = both asset and risk**: 300 commit in 10 giorni calendar è impossibile umanamente. Productivity multiplier founder + AI è ~3-5x dev tradizionale. Asset (cost-to-ship low) ma anche risk (se AI access removed o founder leaves, productivity collapses).
